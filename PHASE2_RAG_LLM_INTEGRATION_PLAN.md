# Phase 2 Plan: Integrate RAG with On-Device LLM

This plan assumes Phase 1 artifacts are ready:

- `tools/rag/out/florence_pack_clean.json`
- `tools/rag/out/florence_embeddings_all-MiniLM-L6-v2.json`

Goal: use retrieval context from these files to ground responses generated by the existing LiteRT LLM flow.

## 0) Demo-first decision

For demo speed and lower integration risk, start with **in-memory RAG** and keep Room as an optional optimization.

- In-memory path: fastest to implement, fewer moving parts.
- Room path: better persistence/startup reuse, but more plumbing.

Because data is static for the demo, in-memory is the recommended first milestone.

## 1) Device artifact placement and load contract

Place both files in app-accessible external files path:

- `/sdcard/Android/data/com.beyondmaps/files/rag/florence_pack_clean.json`
- `/sdcard/Android/data/com.beyondmaps/files/rag/florence_embeddings_all-MiniLM-L6-v2.json`

Add one config source of truth (constants) for:

- destination id (`florence_italy`)
- model name (`all-MiniLM-L6-v2`)
- top-k (start with `k=5`)
- max context chunks (start with 4-6 depending on prompt budget)

## 2) RAG runtime components to implement

Create (or finalize) these components:

1. `VectorPackLoader`
   - Reads both JSON files.
   - Produces:
     - `Map<String, Chunk>` by `chunkId`
     - `List<EmbeddingRow(chunkId, vector)>`
   - Loads once on app startup (background thread).

2. `QueryEmbedder`
   - Embeds user query with the same model dimension (`384`).
   - Validates output vector length.

3. `VectorRetriever`
   - Computes cosine similarity between query vector and all chunk vectors.
   - Returns top-k chunk ids + scores.
   - Applies simple safety filters:
     - destination match
     - non-empty text

4. `VectorPromptBuilder`
   - Builds grounded prompt:
     - system guardrails
     - user query
     - retrieved chunk snippets
   - Adds source ids in context block to support traceability/debugging.

## 3) Chat pipeline integration points

### `BeyondMapsChatbot`

Update request flow:

1. user message
2. query embedding
3. retrieve top-k
4. build augmented prompt
5. send prompt to LiteRT LLM
6. stream/return response

Fallback behavior:

- If any RAG step fails (files missing, parse error, embedding failure), continue with LLM-only response and log warning.

### `ChatViewModel`

Add UI state for:

- RAG ready/loading/error
- retrieval latency
- selected chunk ids (debug-only)

Expose optional debug strings for demo mode:

- `"RAG: k=5, retrieved=5, t=XXms"`

## 4) Memory/performance strategy (important for current file sizes)

Current artifact sizes are large, so avoid repeated parsing:

- Load once and cache in singleton/repository scope.
- Keep vectors in compact arrays (`FloatArray`) rather than nested maps/JSON nodes.
- Keep chunk text map in memory keyed by `chunkId`.
- Do not re-read JSON per query.

Suggested startup behavior:

- app launch -> async warmup
- show "RAG warming up..." until index ready
- disable RAG retrieval until warmup complete

## 5) Optional Room acceleration path (already scaffolded)

There is existing Room code in `app/src/main/java/com/beyondmaps/data/rag`.

If needed later, use this sequence:

1. Ingest JSON into Room once (version/hash guarded).
2. Keep vectors in DB (`EmbeddingEntity`) and chunks in `KnowledgeChunkEntity`.
3. Build an in-memory vector index from Room at startup (or lazy on first query).

Room should be treated as persistence/indexing support, not mandatory for correctness.

## 6) Minimal acceptance criteria for Phase 2 demo

1. App successfully loads both RAG files from device storage.
2. Query embedding dimension is 384 and retrieval returns top-k chunk ids.
3. Prompt includes retrieved context text.
4. LLM output changes when retrieved context changes.
5. Graceful fallback works when RAG files are unavailable.

## 7) Quick test checklist

Use 5 prompt categories:

1. Florence overview question
2. Neighborhood recommendation
3. Museum/attraction query
4. Accommodation-style query
5. Out-of-scope query ("weather in Tokyo") to verify guardrails/fallback behavior

For each run, verify:

- retrieved chunk ids are non-empty
- latency is acceptable
- response is grounded in retrieved context

## 8) Suggested implementation order

1. File constants + loader (`VectorPackLoader`)
2. Query embedding (`QueryEmbedder`)
3. Retriever (`VectorRetriever`)
4. Prompt builder (`VectorPromptBuilder`)
5. Wire into `BeyondMapsChatbot`
6. Expose state in `ChatViewModel`
7. Add fallback + logs
8. Run demo checklist

## 9) Known risks and mitigations

- **Startup lag:** perform async warmup and show status.
- **Memory pressure:** compact vector representation; avoid duplicate structures.
- **Prompt bloat:** cap chunk count and trim chunk text length.
- **File missing on device:** clear UX message + LLM-only fallback.

---

If Phase 2 works end-to-end with this plan, Phase 3 can focus on optimization (Room import pipeline, binary vector format, and retrieval speed-ups).
